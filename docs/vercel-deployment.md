# Vercel Deployment

Each app in this monorepo deploys as a **separate Vercel project**, deployed from the **monorepo root** using the Vercel CLI. The shared `packages/ui-blocks` and `packages/config` packages are bundled at build time — no separate deployment is needed for them.

---

## How It Works

```
Monorepo root
│
├── .vercel/project.json          ← links this repo to a Vercel project
├── vercel.json                   ← active deployment config
├── package.json                  ← root workspace (includes next devDependency)
│
├── apps/
│   └── resort-admin-platform-application-1/
│       ├── vercel.json           ← alternative config (dashboard-only deploys)
│       └── next.config.ts        ← transpilePackages: ["ui-blocks"]
│
└── packages/
    ├── ui-blocks/                ← transpiled and bundled into the app at build time
    └── config/                   ← bundled into the app at build time
```

When `vercel deploy` is run from the repo root, Vercel CLI reads `.vercel/project.json` to identify the project, then reads the root `vercel.json` for build configuration. The full repo is uploaded so all workspace packages are available during the build.

## Configuration Files

### Root `vercel.json`

```json
{
  "installCommand": "pnpm install --frozen-lockfile",
  "buildCommand": "pnpm --filter=resort-admin-platform-application-1 build",
  "outputDirectory": "apps/resort-admin-platform-application-1/.next",
  "framework": "nextjs"
}
```

| Field | Value | Why |
|-------|-------|-----|
| `installCommand` | `pnpm install --frozen-lockfile` | Installs **all** workspace packages (all 7 projects). No `--workspace-root` flag — that restricts install to root-only, skipping app and package dependencies on Vercel's fresh build environment. |
| `buildCommand` | `pnpm --filter=<app-name> build` | Builds only the target app using pnpm's workspace filter. Equivalent to running `next build` from inside the app directory. |
| `outputDirectory` | `apps/<app-name>/.next` | Points the Next.js adapter to the build output. Required because the build runs at repo root, so the `.next` folder is inside the app subdirectory, not at the root. |
| `framework` | `"nextjs"` | **Critical.** Tells `vercel build` to use the Next.js adapter. Without this, Vercel treats the `.next` output as plain static files — no serverless functions are created, no routing config is generated, and every route returns 404. |

### Why `next` is in the root `package.json`

```json
{
  "devDependencies": {
    "next": "16.1.6"
  }
}
```

When `framework: "nextjs"` is set, `vercel build` validates that `next` exists in the `package.json` of the working directory (the monorepo root). Without this entry, the build fails with:

```
Error: No Next.js version detected. Make sure your package.json has "next"
in either "dependencies" or "devDependencies".
```

Adding it at the root satisfies the validator. The version must match what the app uses. It does not conflict with the app's own `next` dependency — pnpm deduplicates across the workspace.

### `.vercel/project.json`

```json
{
  "projectId": "prj_SOCpN8S3mSRdhGtWH3mp7r640WPo",
  "orgId": "team_KNyqRKRsm3HuoMwk6AOvFVck",
  "projectName": "resort-admin-platform-application-1"
}
```

Links the local repo to the Vercel project. Auto-generated by `vercel link` or `vercel pull`. Do not edit manually.

### `apps/resort-admin-platform-application-1/vercel.json`

```json
{
  "installCommand": "cd ../.. && pnpm install --frozen-lockfile",
  "buildCommand": "pnpm build",
  "outputDirectory": ".next",
  "framework": "nextjs"
}
```

Used **only** when deploying via the Vercel dashboard with Root Directory set to `apps/resort-admin-platform-application-1`. Key differences from the root config:
- `installCommand` navigates up two levels to the workspace root before installing, so `workspace:*` links resolve correctly.
- `outputDirectory` is `.next` (relative to the app directory, not the repo root).
- `framework: "nextjs"` works here because Vercel reads the app's own `package.json`, which has `next` in its dependencies — no root-level `next` entry needed.

> **For CLI deployments from the repo root, this file is ignored. The root `vercel.json` takes precedence.**

## What the Next.js Adapter Does

This is the most important part to understand. When `framework: "nextjs"` is set, `vercel build` runs the **Next.js adapter** after your build command finishes. The adapter:

1. Reads the `.next` output from `outputDirectory`
2. **Traces all server-side files** each route depends on
3. **Creates serverless functions** for every page and API route
4. **Generates a routing config** (`/vercel/output/config.json`) mapping URLs to functions
5. **Collects static assets** into `/vercel/output/static/`

Without the adapter (`framework: null` or undetected framework):
- Steps 2–4 are skipped
- Vercel serves the raw `.next` directory as static files
- There are no routing rules, no serverless functions
- **Every route returns 404**, including `/`

You can confirm the adapter ran by looking for these lines in the build log:

```
Detected Next.js version: 16.1.6
...
Traced Next.js server files in: 60ms
Created all serverless functions in: 148ms
Collected static files (public/, static/, .next/static): 26ms
Build Completed in /vercel/output
```

If those lines are absent, the adapter did not run and the deployment will 404.

## Vercel Project Settings (Dashboard)

The Vercel project dashboard settings must match the `vercel.json` approach. These are the correct values:

| Setting | Value |
|---------|-------|
| Root Directory | *(empty — monorepo root)* |
| Framework Preset | Next.js |
| Install Command | `pnpm install --frozen-lockfile` |
| Build Command | `pnpm --filter=resort-admin-platform-application-1 build` |
| Output Directory | `apps/resort-admin-platform-application-1/.next` |

> The `vercel.json` overrides dashboard settings for CLI deployments. For Git-integrated (auto) deployments, the dashboard settings are used. Keep them in sync to avoid divergence.

To update dashboard settings without the UI, use the Vercel REST API:

```bash
curl -X PATCH "https://api.vercel.com/v9/projects/<projectId>?teamId=<teamId>" \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{
    "installCommand": "pnpm install --frozen-lockfile",
    "buildCommand": "pnpm --filter=resort-admin-platform-application-1 build",
    "outputDirectory": "apps/resort-admin-platform-application-1/.next",
    "framework": "nextjs",
    "rootDirectory": null
  }'
```

## Deploying via CLI

### Prerequisites

```bash
# Install Vercel CLI globally
npm i -g vercel

# Authenticate
vercel login
```

### Deploy to Production

```bash
# From the repo root
vercel deploy --prod
```

Full sequence of what happens:

```
vercel deploy --prod
  │
  ├─ reads .vercel/project.json      → identifies the Vercel project
  ├─ reads root vercel.json          → install / build / output / framework config
  ├─ uploads source files            → ~373KB (excludes node_modules, .git, .next)
  │
  ├─ [Vercel cloud] pnpm install     → installs all 7 workspace packages
  ├─ [Vercel cloud] next build       → compiles the app (ui-blocks transpiled inline)
  ├─ [Vercel cloud] Next.js adapter  → traces files, creates serverless functions
  ├─ [Vercel cloud] /vercel/output   → final deployment artifact
  │
  └─ aliases to production URL
```

### Deploy a Preview

```bash
vercel deploy
```

Creates a unique preview URL without touching production.

### Inspect a Deployment

```bash
vercel inspect <deployment-url> --logs
```

### Redeploy

```bash
vercel redeploy <deployment-url>
```

### Manage Environment Variables

```bash
# Add a variable
echo "value" | vercel env add VAR_NAME production

# List all variables
vercel env ls

# Pull variables to local file
vercel pull --yes
```

## Setting Up a New App

To deploy `resort-owner-platform-application-1` or `resort-user-platform-application-1`:

### Step 1 — Update root `vercel.json`

```json
{
  "installCommand": "pnpm install --frozen-lockfile",
  "buildCommand": "pnpm --filter=resort-owner-platform-application-1 build",
  "outputDirectory": "apps/resort-owner-platform-application-1/.next",
  "framework": "nextjs"
}
```

### Step 2 — Link to a new Vercel project

```bash
vercel link
```

Follow the prompts to create or connect a project. This overwrites `.vercel/project.json`.

### Step 3 — Deploy

```bash
vercel deploy --prod
```

> **Note:** Root `vercel.json` and `.vercel/project.json` target one app at a time. To manage multiple apps from the same working directory, update these two files before each deployment or maintain separate checkout directories.

## How Workspace Packages Are Resolved

`ui-blocks` and `@resort/config` are local packages declared with `workspace:*`. They are not published to npm.

```
pnpm install
  └─ links packages/ui-blocks     → node_modules/ui-blocks
  └─ links packages/config        → node_modules/@resort/config
            │
            ▼
next.config.ts: transpilePackages: ["ui-blocks"]
            │
            ▼
Next.js compiles ui-blocks TypeScript source directly at build time
(no pre-build of ui-blocks required)
            │
            ▼
ui-blocks and @resort/config bundled inside the deployed app
No separate Vercel project needed for either package
```

## Troubleshooting

### Every route returns 404 after a successful build

**Cause:** `framework` was set to `null` or not detected. `vercel build` skipped the Next.js adapter and served the raw `.next` folder as static files. No routing config or serverless functions were created.

**How to confirm:** Check the build log for the absence of `Traced Next.js server files` and `Created all serverless functions`. If those lines are missing, the adapter did not run.

**Fix:** Ensure `"framework": "nextjs"` is set in the root `vercel.json` AND that `"next": "<version>"` is present in the root `package.json` devDependencies (matching the app's Next.js version). Run `pnpm install` to update the lockfile, then redeploy.

---

### "No Next.js version detected"

**Cause:** `framework: "nextjs"` is set but the root `package.json` does not have `next` in `dependencies` or `devDependencies`. The `vercel build` validator checks the package.json of the working directory (the monorepo root), not the app subdirectory.

**Fix:** Add `next` to the root `package.json` devDependencies with the same version the app uses, then run `pnpm install` to update the lockfile.

```json
{
  "devDependencies": {
    "next": "16.1.6"
  }
}
```

---

### Build fails — `pnpm install` exits with `ERR_PNPM_NO_MATCHING_VERSION_INSIDE_WORKSPACE`

**Cause:** The deployment's Root Directory is set to the app subdirectory in the Vercel dashboard, so pnpm runs from inside the app folder and cannot find workspace packages (`@resort/config`, `ui-blocks`).

**Fix:** Set the Vercel project's Root Directory to empty (monorepo root) in the dashboard. Or, if keeping the app as Root Directory, prefix the install command with `cd ../.. &&` so pnpm runs from the workspace root.

---

### Build fails — `pnpm install --frozen-lockfile` exits with code 1

**Cause:** `--workspace-root` was appended to the install command. This flag restricts `pnpm install` to only installing root-level dependencies, leaving all app and package dependencies uninstalled.

**Fix:** Remove `--workspace-root`. Use `pnpm install --frozen-lockfile` only.

---

### Dashboard builds fail but CLI builds succeed

**Cause:** The Vercel dashboard project settings (install/build command, framework, root directory) differ from what `vercel.json` specifies. `vercel.json` overrides dashboard settings for CLI deployments only. Git-triggered auto-deploys use the dashboard settings.

**Fix:** Keep dashboard settings in sync with `vercel.json`. Use the REST API patch command above to update all four fields at once.

---

### `workspace:*` packages not found during build

**Cause:** `pnpm install` ran from inside the app directory without access to the repo root where `pnpm-workspace.yaml` is located.

**Fix:** Ensure the install command runs from the monorepo root. For CLI deploys from root this is automatic. For dashboard deploys with app-level Root Directory, use `cd ../.. && pnpm install --frozen-lockfile`.
